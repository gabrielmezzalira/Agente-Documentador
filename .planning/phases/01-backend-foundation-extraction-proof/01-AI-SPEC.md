## 1. System Classification

**System Type:** Extraction

**Description:**
DocuData's extraction pipeline receives project management documents uploaded by gerentes (project managers) at CITi — a student consulting organization managing data science projects. Files arrive in varied formats: plain text task documents, DOCX meeting notes, kanban screenshots, and PDFs. For each upload, the system calls Gemini 2.5 Flash via a LangGraph stateful graph and produces a structured 6-field JSON row (resumo, tarefas, decisoes, problemas, contexto_cliente, proximos_passos) persisted to Supabase. "Good" means every field is populated with content that actually appeared in the source file, correctly categorized, and detailed enough that a gerente reading the extraction six months later would recognize their own sprint without needing to re-read the original document.

**Critical Failure Modes:**
1. **Silent field fabrication:** Gemini invents content for a field — for example, listing a technical decision that was never made — and the gerente treats it as ground truth in a generated retrospective or decision log. Because there is no human review step between extraction and generation, fabricated data propagates into official project documentation.
2. **Cross-field misclassification:** Content lands in the wrong field — a blocker is filed as a "next step," or a client requirement is stored as a "decision." The generation graph compiles context field-by-field; misclassified content is silently omitted from the document type that needs it (e.g., bloqueios missing from a sprint status report).
3. **Empty or near-empty extraction on valid documents:** The graph returns an extraction where most list fields contain zero or one items for a document that clearly had substantial content (a 3-page meeting note). The gerente sees a thin summary and either loses trust in the system or manually reconstructs the sprint record — defeating the tool's entire purpose.
4. **Silent save failure after successful extraction:** Gemini returns a valid structured result but the Supabase insert fails (network error, schema mismatch, expired token). The gerente receives a success indicator but the row is never written. The sprint's data is permanently lost with no retry path.
5. **Truncation without warning on large files:** A long document (project report, multi-sprint summary) is silently truncated at the 50,000-character limit. The extraction reflects only the first portion; later-sprint content is never captured. The gerente has no visibility that this happened.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Internal project management tooling — student consulting organization (non-regulated)

**User Population:** Gerentes (project managers) at CITi's data sub-area. Each gerente manages 1-3 active data science projects with 2-5 member teams. They produce informal working documents throughout a sprint (task documents for analysts, kanban board states, meeting notes, PDFs from client calls) and currently have no standardized documentation practice. Technical familiarity varies; all are university students with some data or software background.

**Stakes Level:** High

**Output Consequence:** Extracted JSON rows are the sole input to the generation graph (Phase 2). Every sprint status report, retrospective, decision log, and complete project document is built exclusively from these rows. If an extraction is wrong, the error propagates invisibly into every downstream document generated for that sprint or project — and because the system replaces manual documentation, there is no parallel written record to catch the error.

### What Domain Experts Evaluate Against

An experienced gerente or subarea coordinator reviewing an extraction would apply these criteria:

---

**Dimension:** Resumo fidelity — does the summary capture what actually happened?

Good (expert accepts): The resumo field names the specific deliverable worked on in that sprint ("Finalizamos o pipeline de ingestao de dados do cliente X e entregamos o modelo de clustering v2"), matches the document's main narrative, and would allow someone unfamiliar with the project to understand the sprint's outcome without reading the original file.

Bad (expert flags): The resumo is generic ("Trabalhamos em diversas tarefas do projeto") or contradicts explicit content in the source document, or describes activities from a different sprint period not mentioned in the file.

Stakes: Critical

Source: Sprint retrospective best-practice literature (Scrum.org, Atlassian) establishes that effective sprint documentation must capture concrete outcomes, not process descriptions. If the summary is wrong, every generated document built from it misrepresents the project history.

---

**Dimension:** Task completeness — are all tasks listed, not just the prominent ones?

Good (expert accepts): The tarefas list includes every discrete work item mentioned in the document, including secondary tasks that appear in passing ("revisao do schema do banco", "alinhamento com cliente sobre entrega"). Partial tasks are listed as partial. A gerente can map each item back to an actual activity.

Bad (expert flags): The list contains only the 2-3 most prominent tasks while omitting 4-5 smaller items mentioned in the document. Or it contains vague items that could apply to any sprint ("fazer analise", "implementar feature") that provide no distinguishing information.

Stakes: High

Source: Document completeness is a core metric for AI document extraction systems (Unframe AI, 2025). In a consulting context, task lists drive billing accountability and performance review — missing tasks create gaps in the record.

---

**Dimension:** Decision capture — are technical decisions recorded with enough context to be useful later?

Good (expert accepts): The decisoes field records the actual decision made ("Decidimos usar Random Forest em vez de XGBoost por conta da interpretabilidade exigida pelo cliente"), not just the topic ("decisao sobre modelo"). Each decision is self-contained — someone reading it 6 months later understands what was decided and the reasoning, without needing to re-read the source.

Bad (expert flags): The field lists discussion topics instead of decisions ("discutimos sobre arquitetura"), or the decisions are too abstract to be actionable ("escolhemos a melhor abordagem"), or decisions from different meetings are conflated into one item.

Stakes: Critical

Source: Decision logs are the highest-value artifact in software project documentation (established PM practice). The generation graph uses the decisoes field to build the project-wide decision log — vague entries make that log useless and undermine the system's core value proposition.

---

**Dimension:** Problem attribution — do the problems field entries describe specific blockers, not general difficulties?

Good (expert accepts): Each problemas entry names the specific issue ("API do cliente retornou dados inconsistentes no campo CPF — bloqueando a etapa de validacao"), who or what it affects, and whether it was resolved. A coordinator reading this can assess project health.

Bad (expert flags): The field contains generic observations ("tivemos dificuldades") or lists technical challenges that were resolved internally without impact on the sprint as blockers. Or the field is empty when the document clearly describes a blocker.

Stakes: High

Source: Sprint status reports and retrospectives structured around specific blockers (not generic difficulties) are the standard advocated by Scrum practitioners (Scrum.org). Blockers that are not recorded as such disappear from the generated sprint status report.

---

**Dimension:** Client context preservation — does the contexto_cliente field carry forward business requirements that are not obvious from task names?

Good (expert accepts): The field captures the client's stated requirements, constraints, or feedback that influenced sprint work ("Cliente exige que o modelo seja explicavel para a equipe juridica — LGPD") even if those requirements are mentioned only once in the document. Someone building the complete project document later will find the business context here.

Bad (expert flags): The field is empty or generic ("cliente quer o projeto bem feito") when the document contains specific client requirements. Or it conflates internal team decisions with client requirements.

Stakes: High

Source: In consulting project management, capturing client-specific constraints is essential for knowledge transfer and audit trail. CITi projects turn over team members each semester; contexto_cliente is the primary vehicle for preserving institutional client knowledge.

### Known Failure Modes in This Domain

1. **Anchoring to document structure rather than content type.** When an informal meeting note lists items in a numbered list, the model may classify all numbered items as tarefas even when some are clearly decisions or problems. The structural cue (numbered list) overrides the semantic signal (blocker description). This is an instance of the order-bias and framing-effects failure documented in LLM extraction research (arxiv 2602.06176).

2. **Over-compression of multi-topic documents.** A document covering two sprints, or a weekly sync that touches multiple projects, produces a single resumo that averages across all topics rather than focusing on the sprint the gerente assigned the upload to. The model loses the sprint-scoping constraint when the document's own structure does not reinforce it.

3. **Kanban image misread — status conflation.** When processing a kanban screenshot (Phase 2), cards in "In Progress" and "Done" columns may be merged into the tarefas list without status distinction, or cards with long titles may be truncated at a visually ambiguous line break. The result is a task list that omits status — making the sprint status report generator unable to distinguish completed from pending work.

4. **Implicit decision extraction failure.** In informal meeting notes, decisions are frequently stated as conclusions without the word "decidimos" (e.g., "Vamos usar o modelo X"). The model may not recognize these as decisions and instead files them as proximos_passos or omits them. Decisions recorded only implicitly in informal language are systematically under-extracted.

### Regulatory / Compliance Context

None identified for this deployment context. CITi is a student organization; DocuData processes internal project management documents only. No personal health data (HIPAA), financial data (FCA/SOX), or EU resident personal data (GDPR) is in scope for the MVP. The documents ingested are work artifacts produced by team members about their own projects.

One internal constraint applies: client names and project details are confidential between CITi and its clients. The system must not log full document content in production (enforced via the 200-character logging limit in `extrair_conteudo`).

## 2. Framework Decision

**Selected Framework:** LangGraph

**Version:** langgraph>=0.2, langchain-google-genai (latest), langchain-core (latest), supabase==2.4.6 (pinned — v1/v2 silent break)

**Rationale:**
The extraction pipeline's defining requirement is conditional retry at a single node (`estruturar_output`) without re-running upstream steps. LangGraph StateGraph is the only framework in the decision matrix that supports this natively via `add_conditional_edges` — every other option requires either re-running the full chain or hand-rolling state management. The TypedDict state schema maps directly to `ExtractionState` from the design doc, `with_structured_output()` on `ChatGoogleGenerativeAI` satisfies D-05 (no JsonOutputParser), and `ainvoke()` satisfies D-06 (non-blocking async). The decision is forward-compatible: Phase 2 adds `detectar_tipo` branching and Phase 3 adds the generation graph — both are additive node/edge changes, not rewrites.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain LCEL chains | Simpler for linear flow but lacks per-node retry and explicit state — refactoring cost in Phase 2 (multi-type branching) outweighs reduced scaffolding |

**Vendor Lock-In Accepted:** Partial — LangGraph is model-agnostic (not LLM vendor lock-in), but ties the architecture to LangChain's ecosystem. Gemini 2.5 Flash is a deliberate provider choice (API key gratuita, vision support) documented in CLAUDE.md §12.

### Entry Point Pattern

This is the minimal working pattern for a LangGraph stateful extraction graph. Compile once at module level (not inside a request handler). Call `await graph.ainvoke()` from async FastAPI endpoints.

```python
# graphs/extraction_graph.py
from typing import TypedDict, Optional
from langgraph.graph import StateGraph, START, END
from langchain_google_genai import ChatGoogleGenerativeAI
from langchain_core.messages import HumanMessage, SystemMessage
from pydantic import BaseModel, Field

# --- 1. Output schema (6-field extraction) ---
class ConteudoEstruturado(BaseModel):
    resumo: str = Field(description="Resumo do que foi trabalhado")
    tarefas: list[str] = Field(description="Tarefas identificadas")
    decisoes: list[str] = Field(description="Decisoes tecnicas tomadas")
    problemas: list[str] = Field(description="Problemas e bloqueios")
    contexto_cliente: str = Field(description="Informacoes sobre o cliente ou requisitos")
    proximos_passos: list[str] = Field(description="Proximos passos identificados")

# --- 2. State schema ---
class ExtractionState(TypedDict):
    arquivo_bytes: bytes
    arquivo_nome: str
    mime_type: str
    sprint_numero: int
    projeto_id: str
    tipo: str                            # "texto" | "imagem" | "pdf"
    texto_preprocessado: str
    conteudo_estruturado: Optional[dict]
    valido: bool
    tentativas: int
    erro: Optional[str]

# --- 3. Model — bind structured output once, reuse across requests ---
_llm = ChatGoogleGenerativeAI(
    model="gemini-2.5-flash",
    temperature=0,
    max_tokens=2048,
)
_structured_llm = _llm.with_structured_output(ConteudoEstruturado)

# --- 4. Node functions (return partial state dicts) ---
def detectar_tipo(state: ExtractionState) -> dict:
    mapping = {
        "text/plain": "texto",
        "application/pdf": "pdf",
        "image/png": "imagem",
        "image/jpeg": "imagem",
        "image/jpg": "imagem",
        "image/webp": "imagem",
    }
    tipo = mapping.get(state["mime_type"], "desconhecido")
    return {"tipo": tipo}

def preprocessar_arquivo(state: ExtractionState) -> dict:
    if state["tipo"] == "texto":
        texto = state["arquivo_bytes"].decode("utf-8")
        return {"texto_preprocessado": texto}
    # PDF / imagem handling added in Phase 2
    return {"texto_preprocessado": state["arquivo_bytes"].decode("utf-8", errors="replace")}

async def extrair_conteudo(state: ExtractionState) -> dict:
    prompt_suffix = ""
    if state["tentativas"] > 0:
        prompt_suffix = " Retorne APENAS o JSON, sem texto antes ou depois, sem markdown."

    messages = [
        SystemMessage(content=(
            "Voce e um extrator de conhecimento de projetos de dados. "
            "Extraia as informacoes do arquivo e retorne no schema solicitado."
            + prompt_suffix
        )),
        HumanMessage(content=state["texto_preprocessado"]),
    ]
    try:
        resultado: ConteudoEstruturado = await _structured_llm.ainvoke(messages)
        return {
            "conteudo_estruturado": resultado.model_dump(),
            "valido": True,
        }
    except Exception as exc:
        return {
            "conteudo_estruturado": None,
            "valido": False,
            "tentativas": state["tentativas"] + 1,
            "erro": str(exc),
        }

def _roteador(state: ExtractionState) -> str:
    if state["valido"]:
        return "salvar"
    if state["tentativas"] < 2:
        return "extrair_conteudo"   # retry
    return END                      # exhaust retries — surface error

async def salvar(state: ExtractionState) -> dict:
    from services.supabase_client import get_client
    client = get_client()
    client.table("ingestions").insert({
        "project_id": state["projeto_id"],
        "sprint_number": state["sprint_numero"],
        "file_name": state["arquivo_nome"],
        "file_type": state["tipo"],
        "extracted_content": state["conteudo_estruturado"],
    }).execute()
    return {}

# --- 5. Build graph — compile ONCE at module level ---
_builder = StateGraph(ExtractionState)
_builder.add_node("detectar_tipo", detectar_tipo)
_builder.add_node("preprocessar_arquivo", preprocessar_arquivo)
_builder.add_node("extrair_conteudo", extrair_conteudo)
_builder.add_node("salvar", salvar)

_builder.add_edge(START, "detectar_tipo")
_builder.add_edge("detectar_tipo", "preprocessar_arquivo")
_builder.add_edge("preprocessar_arquivo", "extrair_conteudo")
_builder.add_conditional_edges("extrair_conteudo", _roteador)
_builder.add_edge("salvar", END)

extraction_graph = _builder.compile()  # <-- module-level singleton

# --- 6. FastAPI usage ---
# from graphs.extraction_graph import extraction_graph
#
```

### Key Abstractions

| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `StateGraph` | The graph builder — defines nodes and edges against a `TypedDict` state | Once per graph, at module level |
| `TypedDict` state | A typed dict that carries all data between nodes. Nodes return partial dicts (only the fields they change) | Define before building the graph; every node receives and returns it |
| `add_conditional_edges` | Adds a routing function that decides the next node based on state — enables retry loops and branching | After any node that can branch (e.g., validation pass/fail, retry) |
| `.compile()` | Finalises the graph into a `CompiledStateGraph` (a `Runnable`) | Exactly once at module level — never inside a request handler |
| `.ainvoke()` | Async execution of the compiled graph; runs all nodes in order per the edges | In async FastAPI endpoint handlers — replaces sync `invoke()` |

### Common Pitfalls

1. **Calling `.compile()` inside a request handler.** Every POST /ingest would rebuild and recompile the graph, wasting 10-50 ms per request and creating a new object each time. Compile once at module load time and import the singleton into your router.

2. **Using sync `graph.invoke()` in an async FastAPI endpoint.** `invoke()` is blocking — it will stall the event loop and prevent other concurrent requests from being served. Always use `await graph.ainvoke()` in async endpoint handlers. If you must use sync code, run it via `asyncio.get_event_loop().run_in_executor()`.

3. **Returning the full state dict from a node instead of a partial dict.** LangGraph merges node return values into the current state. If you return the entire state unchanged, you signal no fields were updated. Only return the keys your node actually modifies.

4. **Expecting `with_structured_output()` to never raise.** If Gemini returns a malformed response, the Pydantic validation inside `with_structured_output()` raises a `ValidationError`. Wrap `_structured_llm.ainvoke()` in try/except, increment `tentativas`, and route back via the conditional edge — do not let the exception propagate uncaught to FastAPI.

5. **Missing `poppler-utils` on Railway for PDF processing (Phase 2).** `pdf2image` calls `pdftoppm` under the hood, which requires `poppler-utils` — a system package not installed by default in Railway's build environment. Add a `nixpacks.toml` now (even in Phase 1) to avoid a Phase 2 deploy surprise: `[phases.setup] nixPkgs = ["poppler_utils"]`.

## 4. Implementation Guidance

**Model Configuration:**

```python
from langchain_google_genai import ChatGoogleGenerativeAI

llm = ChatGoogleGenerativeAI(
    model="gemini-2.5-flash",   # D-04: overrides 1.5 Flash from design doc
    temperature=0,              # deterministic extraction — no creative variation
    max_tokens=2048,            # sufficient for 6-field JSON; avoids unbounded cost
    # GOOGLE_API_KEY read from environment automatically
)
```

Use `temperature=0` for extraction tasks. The model should classify and extract, not generate. Setting `max_tokens=2048` is a hard upper bound — the 6-field JSON rarely exceeds 600 tokens, so this leaves headroom for verbose documents while capping runaway cost.

**Core Pattern:**

The graph follows a linear pipeline with a conditional retry loop on the extraction node:

```
START
  -> detectar_tipo        (pure function, reads mime_type, writes tipo)
  -> preprocessar_arquivo (reads arquivo_bytes, writes texto_preprocessado)
  -> extrair_conteudo     (calls Gemini, writes conteudo_estruturado + valido)
  -> [conditional]
       if valido          -> salvar -> END
       if not valido and tentativas < 2 -> extrair_conteudo (retry)
       if not valido and tentativas >= 2 -> END (surface erro)
```

The conditional edge function (`_roteador`) reads only `state["valido"]` and `state["tentativas"]`. The retry loop feeds back into `extrair_conteudo` with a hardened prompt suffix on the second attempt.

**Tool Use:**

This graph does not use LangGraph tool-calling (no `ToolNode`). The only external calls are:
- `_structured_llm.ainvoke()` — Gemini 2.5 Flash via langchain-google-genai
- `supabase_client.table("ingestions").insert()` — supabase-py v2 sync API

For the Gemini call, `with_structured_output(ConteudoEstruturado)` handles schema enforcement. No manual JSON parsing is needed. Do not use `JsonOutputParser` — it does not validate against a Pydantic model and silently passes malformed output.

**State Management:**

LangGraph state is in-memory per invocation (no checkpointing configured for Phase 1). Each call to `extraction_graph.ainvoke(initial_state)` creates an isolated state dict. Nodes return partial dicts — only the keys they modify. The graph merges returned partial dicts into the running state at each step.

Do not configure a `checkpointer` for Phase 1. Persistence is handled by the explicit `salvar` node writing to Supabase. Adding a LangGraph checkpointer would require a `thread_id` per invocation and a backing store (Postgres/Redis) — unnecessary overhead for a stateless extraction job.

**Context Window Strategy:**

Gemini 2.5 Flash context window is 1M tokens. For Phase 1 (TXT files only), context overflow is not a realistic concern. Apply a defensive pre-processing limit anyway: truncate `texto_preprocessado` to 50,000 characters before sending to Gemini. A meeting note or task document in Portuguese rarely exceeds 5,000 characters; the limit protects against accidental uploads of large log files.

```python
MAX_CHARS = 50_000

def preprocessar_arquivo(state: ExtractionState) -> dict:
    texto = state["arquivo_bytes"].decode("utf-8")
    return {"texto_preprocessado": texto[:MAX_CHARS]}
```

### 4b.1 Structured Outputs with Pydantic

For this LangGraph + Gemini extraction pipeline, `with_structured_output()` is the correct integration pattern. It binds a Pydantic model to the LLM call and handles schema enforcement without manual JSON parsing.

```python
from pydantic import BaseModel, Field
from langchain_google_genai import ChatGoogleGenerativeAI

# Define the 6-field extraction schema as a Pydantic model
class ConteudoEstruturado(BaseModel):
    """Schema de extracao de conhecimento de projetos de dados."""
    resumo: str = Field(description="Descricao do que foi trabalhado nesta entrega")
    tarefas: list[str] = Field(description="Lista de tarefas identificadas no arquivo")
    decisoes: list[str] = Field(description="Decisoes tecnicas tomadas nesta sprint")
    problemas: list[str] = Field(description="Problemas e bloqueios identificados")
    contexto_cliente: str = Field(description="Informacoes sobre o cliente ou requisitos de negocio")
    proximos_passos: list[str] = Field(description="Lista de proximos passos identificados")

# Bind the schema to the model — compile once at module level
llm = ChatGoogleGenerativeAI(model="gemini-2.5-flash", temperature=0, max_tokens=2048)
structured_llm = llm.with_structured_output(ConteudoEstruturado)

# In the extraction node — wrap in try/except and handle retry via state
async def extrair_conteudo(state: ExtractionState) -> dict:
    messages = build_messages(state)          # system + user messages
    try:
        resultado: ConteudoEstruturado = await structured_llm.ainvoke(messages)
        return {
            "conteudo_estruturado": resultado.model_dump(),
            "valido": True,
        }
    except Exception as exc:
        # Log the raw error for debugging — Pydantic ValidationError includes the offending field
        print(f"[extrair_conteudo] attempt {state['tentativas'] + 1} failed: {exc}")
        return {
            "valido": False,
            "tentativas": state["tentativas"] + 1,
            "erro": str(exc),
        }
```

Retry logic: the `_roteador` conditional edge checks `state["valido"]` and `state["tentativas"]`. On failure, it routes back to `extrair_conteudo` (max 2 retries). On the second attempt, the system prompt gains a suffix: `"Retorne APENAS o JSON, sem texto antes ou depois, sem markdown."` On exhausted retries, the graph routes to `END` and the FastAPI router should return HTTP 502 with the `state["erro"]` message.

Log: the exception message, the attempt number, and the first 200 characters of `texto_preprocessado`. Do not log full file content in production.

### 4b.2 Async-First Design

LangGraph fully supports async. Node functions can be `async def` or sync `def` — the framework handles both. Use async node functions for any I/O-bound work (Gemini calls, Supabase inserts).

```python
# CORRECT — async node for I/O work
async def extrair_conteudo(state: ExtractionState) -> dict:
    resultado = await structured_llm.ainvoke(messages)
    return {"conteudo_estruturado": resultado.model_dump(), "valido": True}

# CORRECT — sync node for CPU-bound work (file parsing, MIME detection)
def detectar_tipo(state: ExtractionState) -> dict:
    return {"tipo": mapping.get(state["mime_type"], "desconhecido")}

# FastAPI endpoint — always await ainvoke, never invoke
@router.post("/ingest")
async def ingest(arquivo: UploadFile, sprint_numero: int = Form(...), projeto_id: str = Form(...)):
    state = build_initial_state(arquivo, sprint_numero, projeto_id)
    result = await extraction_graph.ainvoke(state)   # non-blocking
    return {"status": "ok", "sprint": sprint_numero}
```

The one common mistake: calling `asyncio.run()` from inside an async context (e.g., inside an async FastAPI endpoint or an async node function). `asyncio.run()` creates a new event loop and raises `RuntimeError: This event loop is already running`. Inside async code, always use `await`, never `asyncio.run()`.

Stream vs. await: use `await graph.ainvoke()` for extraction — you need the complete, validated structured output before you can write to Supabase. Streaming (`graph.astream()`) is useful for generation flows where you want to return markdown tokens to the frontend progressively. Do not stream extraction results.

### 4b.3 Prompt Engineering Discipline

Always separate system and user prompts. The system prompt sets the role and constraints (never changes between requests). The user prompt carries the document content (changes per request).

```python
from langchain_core.messages import SystemMessage, HumanMessage

SYSTEM_PROMPT = (
    "Voce e um extrator de conhecimento de projetos de dados do CITi. "
    "Dado um arquivo de projeto (ata, task, kanban, relatorio), extraia as "
    "informacoes estruturadas no schema solicitado. "
    "Seja preciso e factual — extraia apenas o que esta explicitamente presente no texto. "
    "Nao infira informacoes que nao estejam no documento."
)

def build_messages(state: ExtractionState, retry: bool = False) -> list:
    system_content = SYSTEM_PROMPT
    if retry:
        system_content += " Retorne APENAS o JSON do schema, sem texto adicional."
    return [
        SystemMessage(content=system_content),
        HumanMessage(content=state["texto_preprocessado"]),
    ]
```

Few-shot examples: for Phase 1 (TXT files with clear structure), few-shot is not needed — the Pydantic schema provides sufficient signal. If extraction quality degrades on noisy documents in Phase 2, add 1-2 inline examples in the system prompt (not dynamic retrieval — the documents are short enough that inline examples stay well within context).

Always set `max_tokens` explicitly. Never leave it unbounded in production — a single runaway call at 1M tokens can exhaust the daily quota. For this extraction task, 2048 tokens is a generous upper bound.

### 4b.4 Context Window Management

Gemini 2.5 Flash supports 1M token context. For this extraction pipeline (Phase 1: TXT files), realistic document sizes are 500-5,000 tokens. Apply a character-level truncation guard before the Gemini call to prevent accidental large uploads from consuming disproportionate quota:

```python
MAX_INPUT_CHARS = 50_000   # ~12,500 tokens — well within window, protects quota

def preprocessar_arquivo(state: ExtractionState) -> dict:
    texto = state["arquivo_bytes"].decode("utf-8")
    if len(texto) > MAX_INPUT_CHARS:
        # Log the truncation so the manager knows context was cut
        print(f"[preprocessar] {state['arquivo_nome']} truncated: {len(texto)} -> {MAX_INPUT_CHARS} chars")
        texto = texto[:MAX_INPUT_CHARS]
    return {"texto_preprocessado": texto}
```

For the generation graph (Phase 3): when compiling context from many ingestions, the `compilar_contexto` node concatenates all sprint data. If a project has 10+ sprints, the compiled context can reach 20,000+ tokens. Apply per-ingestion truncation (`resumo` limited to 500 chars, lists limited to 10 items each) to keep the generation prompt under 30,000 tokens regardless of project age.

### 4b.5 Cost and Latency Budget

**Gemini 2.5 Flash pricing (as of May 2026):** ~$0.075 per 1M input tokens, ~$0.30 per 1M output tokens.

Per-call estimate for extraction:
- Input: ~2,000 tokens (system prompt ~300 + document ~1,700) = $0.00015
- Output: ~400 tokens (6-field JSON) = $0.00012
- Per ingestion call: ~$0.00027 (~$0.0003)
- At 100 ingestions/day: ~$0.03/day — negligible

Retry doubles the cost per failed call. At the 2-retry limit, worst case is 3x per document ($0.0009). Still negligible at expected volume.

**Latency:** Gemini 2.5 Flash p50 latency is ~1-3 seconds for this prompt size. The FastAPI endpoint should set a 30-second timeout on the graph invocation. Expose `tentativas` in the response so the caller knows if a retry occurred.

**Caching:** For Phase 1, exact-match caching is not practical (every document is unique). If in Phase 2 the same kanban screenshot is uploaded twice, a hash-based deduplication check in the `/ingest` router (before invoking the graph) avoids a redundant Gemini call:

```python
import hashlib

file_bytes = await arquivo.read()
file_hash = hashlib.sha256(file_bytes).hexdigest()
existing = client.table("ingestions").select("id").eq("file_hash", file_hash).execute()
if existing.data:
    return {"status": "duplicate", "ingestion_id": existing.data[0]["id"]}
```

Note: `file_hash` column would need to be added to the `ingestions` table schema — flag as a Phase 2 enhancement.

**Model routing for sub-tasks:** The extraction pipeline uses a single Gemini 2.5 Flash call. No sub-task routing is needed in Phase 1. If in Phase 3 the generation graph becomes expensive, route the document-type classification step (`selecionar_prompt`) to a lighter call and reserve Gemini 2.5 Flash for the final generation step.

## 5. Evaluation Strategy

> Phase 1 scope: TXT files only. Evals validate the extraction pipeline end to end (LangGraph -> Gemini 2.5 Flash -> Supabase). MVP posture: practical, not exhaustive — code-based checks carry the load; LLM-judge checks cover the subjective fidelity dimensions the domain experts care about. Calibrate the LLM judge against the human-labeled reference set before trusting its scores.

### Dimensions

| # | Dimension | Rubric (Pass/Fail) | Measurement | Priority |
|---|-----------|--------------------|-------------|----------|
| 1 | **Schema validity** | PASS: extraction returns all 6 fields with correct types — `resumo` and `contexto_cliente` are non-null strings; `tarefas`, `decisoes`, `problemas`, `proximos_passos` are lists of strings (never null, never a bare string, never a dict). FAIL: any field missing, wrong type, or null where a string is required. Maps to failure mode #2 (silent structural drift). | Code — assert against `ConteudoEstruturado` Pydantic model + explicit type checks on the persisted `extracted_content` dict | Critical |
| 2 | **Resumo fidelity** | PASS: `resumo` names the concrete deliverable/outcome of the sprint, is traceable to the source document's main narrative, and contains no claim absent from the source. FAIL: generic boilerplate ("trabalhamos em diversas tarefas"), contradicts the source, or describes a different sprint period (failure modes #1 fabrication, #5 truncation; domain failure mode #2 over-compression). | LLM Judge (rubric-scored, calibrated against gerente labels) with human spot-check | High |
| 3 | **Task completeness** | PASS: `tarefas` includes every discrete work item in the document, including secondary tasks mentioned in passing; items are specific enough to map back to a real activity. FAIL: only the 2-3 prominent tasks captured while 4+ smaller items are dropped, or items are vague filler ("fazer analise"). Maps to failure mode #3 (thin extraction). | LLM Judge (rubric-scored against expected-task list in reference set) with human spot-check | High |
| 4 | **Decision capture** | PASS: each `decisoes` entry records the actual decision and its reasoning, self-contained for a reader 6 months later ("usamos Random Forest por interpretabilidade exigida pelo cliente"); implicit decisions ("vamos usar X") are captured. FAIL: lists topics not decisions ("discutimos arquitetura"), abstract entries ("escolhemos a melhor abordagem"), or omits implicit decisions (domain failure mode #4). | LLM Judge (rubric-scored, calibrated by subarea coordinator) | Medium |
| 5 | **Retry edge correctness** | PASS: on a malformed/parse-failing Gemini response the `_roteador` routes back to `extrair_conteudo`, increments `tentativas`, applies the hardened prompt suffix on attempt 2, and stops at exactly 2 retries (max 3 total attempts) — never loops indefinitely; on exhaustion it routes to `END` and surfaces `erro`. FAIL: retry loops past 2, retries on a valid result, or fails to increment `tentativas`. | Code — unit test the conditional edge with a forced `valido=False` state at `tentativas` 0/1/2; assert routing target each time | High |
| 6 | **Supabase write integrity** | PASS: a successful extraction writes exactly one `ingestions` row with the correct `project_id`, `sprint_number`, `file_name`, `file_type`, and an `extracted_content` jsonb that round-trips to the same 6-field dict; on insert failure the request surfaces an error rather than reporting success. FAIL: no row written despite success response (failure mode #4), wrong `project_id`/`sprint_number`, or jsonb does not match the extracted dict. | Code — integration test against a Supabase test schema; read the row back and deep-compare | Critical |

**Coverage of critical failure modes (Section 1):** #1 fabrication -> dims 2 & 4; #2 misclassification -> dims 1, 3, 4; #3 thin extraction -> dim 3; #4 silent save failure -> dim 6 + online guardrail; #5 truncation -> dim 2 + offline truncation-rate metric.

## 6. Guardrails

> MVP posture: this is an internal, batch-use tool for student PMs — no real-time user-facing safety surface, so online guardrails are minimal. The one online check defends the most catastrophic, irreversible failure: data silently lost or saved empty. Quality lives in the offline flywheel.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **6-field completeness check before save** | After `extrair_conteudo`, before `salvar`: any of the 6 fields is null, or both string fields (`resumo`, `contexto_cliente`) are empty/whitespace, or all four list fields are empty. | Block the insert; route to retry if `tentativas < 2`, else return HTTP 502 with `erro`. Prevents an empty/partial row being silently persisted (failure modes #2, #3, #4). |
| **Supabase write confirmation** | `salvar` insert raises or returns no inserted row. | Block the success response; return HTTP 502 so the gerente knows the sprint data was NOT saved and can re-upload. Directly closes failure mode #4 (silent save failure). |
| **Truncation flag** | `preprocessar_arquivo` cuts input at 50,000 chars. | Do not block — set a `truncado=true` flag carried into the response and the trace so the gerente sees that later content was not processed (failure mode #5). |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Resumo fidelity / task completeness / decision-capture pass rate** (LLM judge) | Weekly batch over a sample of production ingestions, weighted toward traces that retried, were truncated, or produced near-empty extractions. | If pass rate drops below baseline, inspect failing traces in LangSmith, refine the system prompt or add a few-shot example, re-run the reference set before shipping the prompt change. |
| **Empty-field rate per field** | All ingestions in the batch, broken down by field. | A spike in empty `decisoes`/`problemas`/`contexto_cliente` signals the implicit-decision or client-context under-extraction failure modes — feed offending docs into the reference set and tune the prompt. |
| **Reference-set regression** | On every extraction-graph or prompt change. | Any reference example that newly fails blocks the prompt change until resolved (regression gate once the judge is calibrated). |

## 7. Production Monitoring

**Tracing Tool:** LangSmith (project `docudata-extraction-phase1`). Native LangGraph instrumentation captures every `/ingest` invocation as a full graph trace: per-node inputs/outputs, the Gemini call (prompt, tokens, latency), the retry loop, and the final state. No additional instrumentation code required — `LANGCHAIN_TRACING_V2=true` is sufficient. Respect the confidentiality constraint: do not attach full document content to traces in production; cap any logged source text at the first 200 characters.

**Key Metrics to Track:**
- **Extraction success rate** — fraction of `/ingest` calls that reach `salvar` and write a row (target > 95%). The headline health metric for failure modes #2/#3/#4.
- **Retry rate** — fraction of invocations where `tentativas >= 1`. A rising retry rate is the earliest signal of model/prompt drift or a Gemini API regression.
- **Latency p50 / p95** — end-to-end graph latency (expect p50 ~1-3s; p95 should stay under the 30s endpoint timeout).
- **Empty-field rate per field** — share of saved rows where each of the 6 fields is empty. Distinguishes "genuinely thin document" from systematic under-extraction (esp. `decisoes`, `contexto_cliente`).
- **Truncation rate** — share of ingestions where `truncado=true` (failure mode #5 visibility).

**Alert Thresholds:**
- Extraction success rate < 90% over a rolling 1-hour window -> alert (likely Gemini outage, Supabase auth/schema failure, or prompt break).
- Retry rate > 25% over a rolling 1-hour window -> alert (model/prompt regression).
- Latency p95 > 20s -> alert (approaching the 30s timeout — Gemini latency degradation).
- Any Supabase insert error -> immediate alert (irreversible data loss risk — failure mode #4).

**Smart Sampling Strategy:** Weight human review (by the new-gerente reviewer, who represents the downstream reader) toward the highest-signal traces, not a uniform sample:
- All invocations that retried (`tentativas >= 1`).
- All truncated invocations (`truncado=true`).
- Near-empty extractions (resumo under ~40 chars or all list fields empty) on documents over ~1,000 chars — the thin-extraction failure mode.
- A small uniform random sample (~5%) of clean successes as a control to catch failures the signals miss (signal-metric divergence).
